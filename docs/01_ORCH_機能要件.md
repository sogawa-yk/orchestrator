# ORCH（Orchestrator Agent）機能要件（詳細版）

## 1. 目的

ORCH は OCI 運用マルチエージェントシステムの中枢であり、ユーザーとの対話入口、シナリオ判定、下位エージェント呼び出し、結果統合、承認仲介を担当する。

PoC では、月次のテナント整理 / リソース棚卸しを成立させるための統括エージェントとして設計する。

本書を含む要求・設計・実装関連文書は、すべて日本語で管理する。

---

## 2. 役割

### 2.1 主責務
- ユーザーとの自然言語会話
- 意図理解とシナリオ判定
- Task 分解とエージェント選定
- 実行順制御
- 複数エージェント結果の統合
- 表示形式の切り替え
- 承認 / 却下 / 修正指示の仲介
- 次アクション提示

### 2.2 非責務
- OCI API への直接アクセス
- コスト計算の実行
- 直接の通知送信
- 直接のリソース変更
- 直接の権限付与

---

## 3. Phase 対応

| Phase | 主な役割 |
|---|---|
| Phase 1 | 月次の棚卸しフロー統括（RI + CQ + NF） |
| Phase 2 | IAM / ENV の追加シナリオ統括 |
| Phase 3 | IP / TA を含む複合フロー統括 |

---

## 4. 主シナリオ（Phase 1）

### 4.1 起点
ユーザーが「今月のテナント整理をしたい」「コストと service limit の観点で確認したい」「整理候補を見たい」などの自然言語要求を入力する。

### 4.2 ORCH の期待動作
1. 発話から対象シナリオが S1（棚卸し）であると判断
2. 期間、スコープ、優先観点の不足情報を補う
3. RI に事実収集を依頼
4. CQ に評価を依頼
5. 結果を統合して上位 3〜5 件を要約表示
6. 深掘り候補を提示
7. 必要に応じて NF にメール案作成を依頼
8. 宛先メールアドレスをユーザーに確認
9. ユーザー承認後、NF に送信を指示
10. 送信結果をまとめて返す

---

## 5. 機能要件

### ORCH-F01 会話入口
- ユーザーからの自然言語要求を受け付けること
- 単発の指示だけでなく、対話継続に対応すること
- 前の発話で確定した前提（対象期間、観点、対象リソース）を会話中に保持できること

### ORCH-F02 シナリオ判定
- 入力発話を S1 / S2-A / S2-B / S3 / S4 または複合に分類できること
- Phase 1 では少なくとも S1 を高精度に扱うこと
- 不足情報があれば最小限の追質問で補完すること

### ORCH-F03 タスク分解
- シナリオごとに必要なエージェント組み合わせを決定できること
- 依存関係のあるタスクを順序制御できること
- 並列実行可能なタスクは並列に起動できること

### ORCH-F04 中間結果保持
- 下位エージェントからの結果を内部状態として保持できること
- 保持結果は issue / resource / owner / compartment などのビューに変換可能であること
- 会話の途中で表示粒度を切り替えられること

### ORCH-F05 結果統合
- RI の事実収集結果と CQ の評価結果を結合し、ユーザー向けの要対応項目一覧を生成できること
- 同一リソースに対する複数の問題を適切にまとめられること
- 根拠を簡潔に説明できること

### ORCH-F06 初回応答の最適化
- 初回応答では、上位 3〜5 件の要対応項目を優先表示すること
- 各項目について「何が問題か」「なぜ優先か」「次に何をできるか」を示すこと
- 同時に、追加で選べる深掘り候補を提示すること

### ORCH-F07 柔軟な表示
- ユーザー要求に応じて issue view / resource view / owner view / cost view / limit view / compartment view に切り替えられること
- 「このリソースを詳しく」「誰に連絡が必要？」のようなフォローアップに応答できること

### ORCH-F08 承認仲介
- NF / IAM / IP など承認実行エージェントから `input_required` を受け取れること
- 承認内容、却下、修正指示をユーザーに提示できること
- ユーザーからの approve / reject / 修正指示を受け、対象エージェントへ伝達できること

### ORCH-F09 通知宛先入力の受理
- PoC では、通知先メールアドレスをユーザーから入力してもらう方式をサポートすること
- 入力されたメールアドレスを通知タスクのパラメータとして NF に渡せること

### ORCH-F10 フォールバック
- 下位エージェントの一部失敗時にも、取得済み結果を元に部分的な応答を返せること
- 失敗した観点と成功した観点を明示できること

---

## 6. 入力と出力

### 6.1 入力
- ユーザー自然言語メッセージ
- 下位エージェントからの task status 更新
- 承認対象 artifact
- ユーザーによる承認 / 却下 / 修正指示
- ユーザー入力のメールアドレス

### 6.2 出力
- ユーザー向け統合応答
- 下位エージェントへの task message
- 承認 UI に相当する確認メッセージ
- 最終完了メッセージ

---

## 6.3 品質保証要件

- すべての変更はローカル環境で単体テストを実施すること
- すべての変更はローカル環境で結合テストを実施すること
- ローカル検証後は Kubernetes クラスタ上の検証環境へ配備すること
- 配備後は Playwright を用いたブラウザ操作ベースの E2E テストを 20 シナリオ以上実施すること
- E2E シナリオには正常系、異常系、承認フロー、主要導線回帰を含めること

## 7. Phase 1 タスクテンプレート

### 7.1 棚卸し開始
```json
{
  "scenario": "S1",
  "time_range": {"start": "2026-03-01", "end": "2026-03-31"},
  "scope": "tenancy",
  "focus": ["cost", "limit", "inventory"]
}
```

### 7.2 RI への委譲
```json
{
  "task_type": "inventory_fact_collection",
  "time_range": {"start": "2026-03-01", "end": "2026-03-31"},
  "resource_types": ["all"],
  "include_dependency": true,
  "include_creator_lookup": true
}
```

### 7.3 CQ への委譲
```json
{
  "task_type": "monthly_inventory_evaluation",
  "time_range": {"start": "2026-03-01", "end": "2026-03-31"},
  "focus": ["cost", "limit", "inventory"],
  "ri_result_ref": "task://ri/123"
}
```

### 7.4 NF への委譲
```json
{
  "task_type": "notification_draft",
  "channel": "email",
  "purpose": "monthly_inventory_followup",
  "recipients": ["user@example.com"],
  "issues": ["ISSUE-001", "ISSUE-003"]
}
```

## 8. 会話フロー（Phase 1）

### 8.1 基本フロー
1. ユーザー: 今月のテナント整理をしたい
2. ORCH: 対象期間や重点観点を補完
3. ORCH → RI: 事実収集依頼
4. ORCH → CQ: 評価依頼
5. RI → ORCH: リソース一覧、作成者候補、依存関係
6. CQ → ORCH: 優先度付き issue 一覧
7. ORCH → ユーザー: 上位 3〜5 件の要約 + 深掘り候補提示
8. ユーザー: この 2 件について連絡したい
9. ORCH: 宛先メールアドレス入力を依頼
10. ユーザー: メールアドレス入力
11. ORCH → NF: 文面作成依頼
12. NF → ORCH: プレビュー返却
13. ORCH → ユーザー: 送信してよいか確認
14. ユーザー: 承認
15. ORCH → NF: approve
16. NF → ORCH: 送信完了
17. ORCH → ユーザー: 結果報告

### 8.2 深掘りフローの例
- 「コスト起点で並べ替えて」
- 「このコンパートメントだけ見せて」
- 「このリソースの根拠を見せて」
- 「誰に連絡が必要？」
- 「全件を見せて」

## 9. 状態管理要件

### 9.1 会話状態
ORCH は少なくとも以下をセッション内で保持する。

- 現在シナリオ
- 対象期間
- 対象スコープ
- RI 結果参照
- CQ 結果参照
- 現在の表示ビュー
- 選択済み issue / resource
- 通知先メールアドレス
- 承認待ち task

### 9.2 タスク状態
- queued
- working
- input_required
- completed
- failed

## 10. ルーティング要件（将来）

### 10.1 Phase 1
Phase 1 は RI / CQ / NF / IAM が共通基盤系として単一系統で存在する前提のため、複雑なスコープ選択は不要。

### 10.2 Phase 2 以降
ENV / TA / IP など同種エージェントが複数存在し得る。  
ORCH は以下を分離して判断する。

1. 必要な skill の選択
2. scope metadata に基づく対象インスタンスの選択

### 10.3 軽量 discovery 方針
- 初期は K8s service discovery を使う
- routing 用 metadata は annotation / CRD / ConfigMap で補完する
- フル機能の専用レジストリは必須としない

---

## 11. エラーハンドリング

### ORCH-E01
RI が失敗した場合、CQ の評価に必要な情報が不足することを明示し、取得できた範囲のみ返す

### ORCH-E02
CQ が失敗した場合、事実一覧は返すが優先度は限定的であると明示する

### ORCH-E03
NF が失敗した場合、送信できなかったことを明示し、文面プレビューまでは保持する

### ORCH-E04
ユーザーが承認を保留した場合、送信前状態のまま終了できる

---

## 12. 非機能要件

### ORCH-NF01 応答性
- 初回要約は過度に冗長にせず、意思決定に必要な上位項目を先に返すこと

### ORCH-NF02 説明可能性
- なぜその issue を上位表示したかを短く説明できること

### ORCH-NF03 監査可能性
- どのエージェントへ何を委譲したかを trace として残せること

### ORCH-NF04 安全性
- 承認前に外部変更系タスクを実行しないこと
