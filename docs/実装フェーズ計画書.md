# マルチエージェント実装フェーズ計画書（大枠）

## 1. 方針

実装は **縦切り（end-to-end）優先** で進める。  
つまり、各エージェントを個別に完成させてから結合するのではなく、まず最小限の会話から結果返却までを通し、その後に各エージェントを段階的に強化する。

関連する仕様書、計画書、タスクリスト、運用手順書はすべて日本語で管理する。

### 基本方針
- 最初に **PoC 主シナリオ** を成立させる
- 主シナリオは **月次のテナント整理**
- 最初の対象エージェントは **ORCH / RI / CQ / NF**
- **IAM / ENV / TA / IP** は将来拡張として後続フェーズで強化
- すべてのエージェントは単体 UI（Chainlit）でも最低限動作可能にする
- データストアは共通基盤として Oracle Database を使うが、エージェントごとに最適な論理モデルを採用する
- すべての変更でローカル単体テストとローカル結合テストを完了する
- Kubernetes クラスタへ配備後、Playwright によるブラウザ E2E を 20 シナリオ以上実施する

---

## 2. 実装順序（推奨）

### 優先順
1. ORCH
2. RI
3. CQ
4. NF
5. IAM
6. ENV
7. TA
8. IP

### 理由
- ORCH が全体の入口であり、他エージェントをつなぐ中核になるため
- RI は Phase 1 主シナリオの起点である「事実収集」を担うため
- CQ は RI の出力を受けて評価を行い、PoC の価値を出すため
- NF をつなぐことで、「抽出 → 評価 → 通知」が一気通貫になるため
- IAM / ENV / TA / IP は Phase 1 必須ではなく、後続の拡張テーマとして段階投入した方が安全なため

## 3. 全体フェーズ

### 完了条件

- 最小のエージェント雛形を複製して新規エージェントを作れる
- 共通基盤でローカル単体テスト、ローカル結合テスト、Kubernetes 配備手順、Playwright 実行基盤を提供できる

## Phase 1: 最小 end-to-end PoC

### 目的
最小のマルチエージェント体験を成立させる。

### 対象エージェント
- ORCH
- RI
- CQ
- NF

### 実現するシナリオ
ユーザーが ORCH に「今月のテナント整理をしたい」と依頼すると、

1. ORCH が RI に棚卸しを依頼
2. RI がリソース一覧と作成者候補を返す
3. ORCH が CQ に評価を依頼
4. CQ が cost / limit / cleanup の issue を返す
5. ORCH が上位 3〜5 件を要約表示
6. ユーザーが通知を選ぶ
7. ORCH が NF にドラフト生成を依頼
8. ユーザーがメールアドレスを入力
9. ORCH が承認を受ける
10. NF がメール送信

### このフェーズでの実装レベル
#### ORCH
- Chainlit UI
- セッション管理
- task 状態管理
- RI / CQ / NF 呼び出し
- 要約生成
- 承認 UI

#### RI
- 最小のリソース棚卸し
- リソース種別 / コンパートメント / リージョン一覧
- 作成者候補の最低限推定
- ORCH に返す JSON スナップショット

#### CQ
- 最低限のルールベース評価
- issue_type
  - cost_spike
  - limit_near_exhaustion
  - cleanup_candidate
- 優先度付け
- 推奨アクション生成

#### NF
- ドラフト生成
- メール送信
- 送信履歴保存

### 完了条件
- ORCH から 3 エージェント呼び出しが end-to-end で通る
- 上位 issue が表示される
- メールドラフト生成と承認後送信ができる
- ローカル単体テストとローカル結合テストが成功している
- Kubernetes 検証環境で Playwright E2E 20 シナリオ以上が成功している

## 6. 実装上の注意

- 最初から各エージェントを作り込みすぎない
- まず end-to-end を通す
- DB 設計は最初から完璧を狙わず、Phase 2 で整える
- LLM 主導部分は必ずガードを入れる
- 変更系エージェントは承認境界を先に固定する
- PoC では「実現性」と「見せ方」の両立を優先する

---

## 7. 結論

最もよい進め方は、

**ORCH → RI → CQ → NF の細い縦切りを先に完成させ、  
その後に IAM → ENV → TA → IP を段階追加する**

という順序である。

この順序により、
- PoC の価値を早期に確認できる
- マルチエージェントの良さを最短で見せられる
- 将来拡張にも無理なくつなげられる
- Converged Database の使い分けも段階的に説明できる
