# タスク一覧: [FEATURE NAME]

**入力**: `/specs/[###-feature-name]/` 配下の設計ドキュメント  
**前提資料**: `plan.md`、`spec.md`、`research.md`、`data-model.md`、`contracts/`

**テスト方針**: テストタスクは任意ではない。すべての機能でローカル単体テスト、
ローカル結合テスト、Kubernetes 配備、Playwright E2E 20 シナリオ以上を含めること。

**構成方針**: タスクはユーザーストーリー単位でグルーピングし、各ストーリーが独立して
実装・検証できるようにする。タスク記述は日本語で行う。

## 形式: `[ID] [P?] [Story] 説明`

- **[P]**: 並列実行可能（別ファイルで依存なし）
- **[Story]**: 対象ユーザーストーリー（例: US1, US2, US3）
- 説明には必ず正確なファイルパスを含める

## パス規約

- **単一プロジェクト**: `src/`, `tests/` をリポジトリ直下に置く
- **Web アプリ**: `backend/src/`, `frontend/src/`, `playwright/`
- **モバイル**: `api/src/`, `ios/` または `android/`
- 下記の例は単一プロジェクト前提。実際には `plan.md` の構成に合わせて調整する

## 依存関係と実行順序

### フェーズ依存

- **セットアップ (Phase 1)**: 依存なし
- **基盤整備 (Phase 2)**: セットアップ完了後。全ユーザーストーリーをブロックする
- **ユーザーストーリー (Phase 3+)**: 基盤整備完了後に開始
- **仕上げ (最終 Phase)**: 対象ユーザーストーリー完了後に開始

### ユーザーストーリー依存

- **User Story 1 (P1)**: Phase 2 完了後に開始。ほかのストーリーに依存しない
- **User Story 2 (P2)**: Phase 2 完了後に開始。US1 と連携しても独立検証可能であること
- **User Story 3 (P3)**: Phase 2 完了後に開始。US1 / US2 と連携しても独立検証可能であること

### 各ユーザーストーリー内の順序

- テストケースを先に定義し、必要に応じて失敗確認を行う
- モデル実装の後にサービス実装、その後にエンドポイントや UI 実装を行う
- ローカル単体テストと結合テスト完了後に Kubernetes へ配備する
- Kubernetes 配備後に Playwright E2E を実行する
- ストーリー完了前に次優先度へ進まない

### 並列実行の機会

- Phase 1 と Phase 2 の [P] タスクは並列実行できる
- 各ストーリー内の単体テストと結合テストは並列実行できる
- モデル作成タスクは別ファイルなら並列実行できる
- 基盤完了後は複数ストーリーを担当者ごとに並列実行できる

## 実装戦略

### MVP 優先（User Story 1 のみ）

1. Phase 1 を完了する
2. Phase 2 を完了する
3. Phase 3 を完了する
4. ローカル単体 / 結合テストを実行する
5. Kubernetes 検証環境へ配備し、Playwright E2E を実行する
6. User Story 1 を独立検証してから次へ進む

### 段階的デリバリー

1. セットアップと基盤整備を完了する
2. User Story 1 を追加し、ローカル検証と Kubernetes 上の E2E を完了する
3. User Story 2 を追加し、同じ検証を繰り返す
4. User Story 3 を追加し、同じ検証を繰り返す
5. 各ストーリーは前ストーリーを壊さずに価値を追加する

### 並列チーム戦略

1. チームで Phase 1 と Phase 2 を完了する
2. その後は担当を分けて User Story 1 / 2 / 3 を並列に進める
3. ただし各担当はローカル検証、Kubernetes 配備、Playwright E2E まで責任を持つ

## 注意事項

- [P] は別ファイル・依存なしの並列可能タスクを示す
- [Story] はトレーサビリティのため対応するユーザーストーリーを示す
- すべてのユーザーストーリーは独立して完了・検証可能でなければならない
- 曖昧なタスク、同一ファイル競合、独立性を壊す依存は避ける
- ローカルテストや E2E を省略するタスク分解は不可
